# Clinical Data Lakehouse — Synthea → Medallion → SDTM

An end-to-end clinical data engineering project: a **medallion-architecture lakehouse**
built on **Databricks** over **Synthea** synthetic patient data, mapped into **CDISC
SDTM** domains, and validated with the open-source **CDISC CORE** conformance engine.

```
Synthea CSV  ──►  BRONZE  ──►  SILVER  ──►  GOLD  ──►  SDTM  ──►  CDISC CORE
 (raw EHR)       raw Delta    typed +      patient    RWD→SDTMIG   conformance
                 (untouched)  conformed +  feature &  v3.4 domains  validation
                              HIPAA Safe   cohort     (DM/VS/LB/    report
                              Harbor de-id tables     CM/PR/MH)
```

The project demonstrates the engineering *and* the clinical-domain judgment that the data
itself doesn't show: which "numbers" are actually codes, why a duration isn't a date under
HIPAA Safe Harbor, why protected attributes stay out of an ML feature table, and how
real-world data is honestly mapped — not faked — into a clinical-trial submission standard.

> **Synthetic data only.** No real PHI. All patients are generated by
> [Synthea](https://github.com/synthetichealth/synthea) and are statistically realistic
> but entirely fictional, so the data carries no privacy, cost, or use restrictions.

## Reproducing the data

The raw data is **not committed** (it's regeneratable). Generate the dataset with a fixed seed:

```bash
# Requires JDK 17+
java -jar synthea-with-dependencies.jar -p 1000 -s 12345 --exporter.csv.export=true
```

The same seed always produces the same dataset — reproducible data sourcing. Output lands
in `output/csv/`; six domains feed the pipeline: `patients`, `encounters`, `conditions`,
`medications`, `observations`, `procedures`.

> **Scale note.** The pipeline is parameterized by `-p`. The committed SDTM reference export
> under [`data/`](data/) is a **100-patient** sample (small enough to browse on GitHub); the
> [ML/analytics layer](ml/) uses **1,000 patients** so the model has enough signal — at that
> scale the Safe Harbor `90+` cohort also has members. Nothing downstream cares about the count.

## The layers

### 🥉 Bronze — raw ingestion
Each source CSV is landed as a Delta table **as-is**, read entirely as strings
(`inferSchema=false`) so nothing is lost or mis-typed on the way in. The only additions are
lineage columns (`_ingested_at`, `_source_file`). Bronze is append-only and immutable — the
replayable source of truth.

### 🥈 Silver — typed, conformed, de-identified
Strings become real types, names are conformed to `snake_case`, foreign keys are made
explicit, and the data is **de-identified**. Deliberate choices that show domain literacy:

- **ZIP and FIPS stay strings** — casting them to integers would destroy leading zeros
  (`02108` → `2108`) and corrupt geographic codes.
- **SNOMED / LOINC / RxNorm codes stay strings** — they're identifiers, not quantities.
- **Mixed observation values are split** — `value_raw` keeps the verbatim result, a
  `try_cast` produces `value_numeric` (null for text findings like *"Never smoked tobacco"*).
- **Durations are derived before dates are dropped** — length-of-stay is computed from the
  raw timestamps, then only the year survives (see de-identification below).

### 🏅 Gold — consumer-shaped tables
Six Silver tables collapse to **one row per patient** via a shared aggregation base, then
branch into two tables for two consumers:

- **`gold.patient_features`** — wide, all-numeric, model-ready (could be handed straight to
  `scikit-learn`/AutoML). Protected attributes (race, ethnicity) are **deliberately
  excluded** to avoid baking bias into a cost/risk model.
- **`gold.patient_cohort`** — human-readable dimensions + measures (age bands, totals) for
  a BI dashboard or cohort analysis.

### 🧬 SDTM — real-world data → CDISC submission standard
The Silver tables are mapped into **SDTM-conformant domains**, one per general observation
class. This is an **honest RWD → SDTM mapping** (the recognized pattern for external control
arms and real-world evidence), *not* a claim that Synthea is trial data:

| Domain | Observation class | Source | Records |
|--------|-------------------|--------|--------:|
| **DM** | Special Purpose (Demographics) | patients | 112 |
| **VS** | Findings (Vital Signs) | observations | 13,875 |
| **LB** | Findings (Laboratory) | observations | 30,470 |
| **CM** | Interventions (Concomitant Meds) | medications | 4,096 |
| **PR** | Interventions (Procedures) | procedures | 17,002 |
| **MH** | Events (Medical History) | conditions | 3,975 |

CDISC Controlled Terminology is applied to `SEX`/`RACE`/`ETHNIC` and the common vitals test
codes; year-only `--DTC` dates are valid partial ISO 8601, so the Safe Harbor output
composes cleanly with SDTM date rules.

## De-identification (HIPAA Safe Harbor)

`silver.patients` is de-identified using the **Safe Harbor** method (45 CFR §164.514(b)(2)):

- **Direct identifiers removed** — names, SSN, driver's license, passport, street address,
  birthplace.
- **Geography** — reduced to state + 3-digit ZIP; the 17 HHS-restricted ZIP3 prefixes
  (areas ≤ 20,000 people) are masked to `000`; precise geocodes (lat/lon), city, county,
  and FIPS are dropped.
- **Dates** — birth and death reduced to **year only**; ages > 89 aggregated to a single
  `90+` category with birth year suppressed for that cohort.
- **Retained** — age (< 90), gender, race, ethnicity, marital status: not identifiers under
  Safe Harbor, and analytically useful.

The synthetic Synthea `patient_id` (a UUID, not derived from any real-world attribute) is
kept as a surrogate join key.

## Validation (CDISC CORE)

The six SDTM domains were exported to **SAS Transport v5 (`.xpt`)** — the FDA submission
format — and run through the [CDISC CORE](https://github.com/cdisc-org/cdisc-rules-engine)
engine against the full **SDTMIG v3.4** rule set:

> **118 rules passed · 14 raised issues · 296 skipped** (run in ~59s)

The ~150K individual findings collapse to **~6 root causes**, triaged into *expected
RWD-mapping artifacts*, *cosmetic metadata gaps*, and *genuine work* (the LOINC lab
long-tail). A spotless pass on real-world data would actually be suspicious — it would mean
the mappings were invented. **[See `VALIDATION.md`](VALIDATION.md)** for the full triage.

The report itself is committed at [`validation/sdtmig-3-4-report.xlsx`](validation/sdtmig-3-4-report.xlsx).

## Analytics & ML — diabetes phenotype prediction

A downstream scikit-learn layer ([`ml/`](ml/)) predicts **type-2 diabetes** from demographics
and general healthcare utilization — a care-management *risk-stratification* framing built
with a deliberately **leakage-aware** design (prediabetes held out of the target; condition
counts, medication count, and protected attributes excluded from the features).

| Model | Test AUC | Recall | Precision |
|-------|:--------:|:------:|:---------:|
| Logistic Regression | 0.949 | 0.881 | 0.627 |
| Gradient Boosting | **0.953** | 0.738 | 0.838 |

The dominant feature is `observation_count` — the model detects diabetes by its **lab-monitoring
footprint**, an interpretable and defensible signal. It runs fully offline (reproduces the gold
feature logic in pandas). **[See `ml/README.md`](ml/README.md)** for the design rationale,
results, and figures.

## Running it yourself

1. **Generate data** with the Synthea command above and upload the six CSVs to a Databricks
   volume (`/Volumes/workspace/bronze/raw`).
2. **Run the notebook** top to bottom — it builds the `bronze`, `silver`, `gold`, and `sdtm`
   schemas in Unity Catalog and exports the SDTM domains.
3. **Validate** the exported domains locally:
   ```bash
   pip install pandas pyreadstat
   python scripts/convert_to_xpt.py
   ./core validate -s sdtmig -v 3-4 -d data/sdtm/xpt
   ```

## Caveats (stated honestly)

- **Synthetic data.** Any pattern here (e.g. cost by demographic) is an artifact of
  Synthea's generation model, not evidence about real-world care. The project demonstrates
  *capability*, not findings.
- **RWD ≠ trial data.** SDTM is built for collected trial data; this is an honest mapping of
  real-world/EHR-shaped data into SDTM domains, following CDISC's observational-study
  guidance — explicitly *not* a claim that Synthea output is a clinical trial.
- **Documented mapping substitutions** (e.g. SNOMED/RxNorm codes standing in for
  MedDRA/WHODrug, which require licenses) are called out in `VALIDATION.md` rather than
  hidden — the way a real Study Data Reviewer's Guide would.
